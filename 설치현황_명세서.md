## 1. 프로젝트 개요

- **목적**: 현대모비스 **59개소 사업장**에 **A, B, C 3개 품목**을 설치하는 현황을,
  여러 설치업체 담당자와 마스터 관리자가 **하나의 웹페이지에서 함께 입력·조회**한다.
- **사용자 유형**
  1. **마스터 관리자 (별도 존재)**: 59개소 **전체를 조회하고 모든 항목을 수정** 가능.
     전용 마스터 링크(마스터 토큰)를 가진다.
  2. **설치업체 담당자**: 자기가 맡은 사업장만 수정 가능. 나머지 사업장은 **조회만** 가능.
     - 업체는 총 **4~6곳**이며, 이들이 59개소를 나눠 맡는다.
     - 한 업체가 **여러 사업장**을 맡음. (권한의 단위 = "사업장"이 아니라 "업체")
- **배포 형태**: 링크 하나로 접속. 업체별 고유 링크 + 마스터 링크를 배포.
- **권한 목적**: "남의 사업장을 실수로 수정하는 것"을 막는 수준.
  (아래 RLS + RPC로 실제 서버단 강제를 건다.)

| 역할 | 사용 기술 | 비고 |
|------|-----------|------|

## 3. 데이터 모델 (Supabase 테이블)

아래 SQL을 Supabase의 SQL Editor에 그대로 실행하면 테이블이 만들어진다.

### 3-1. contractors (설치업체 = 권한 단위, 4~6곳)

```sql
create table contractors (
  id            bigint generated always as identity primary key,
  name          text not null,                          -- 업체명
  access_token  text unique not null default gen_random_uuid()::text,  -- 업체별 고유 토큰
  created_at    timestamptz default now()
);
```

- `access_token` 이 곧 그 업체의 접속 열쇠. 링크는 `...?token=<access_token>` 형태.

### 3-2. app_admins (마스터 관리자)

```sql
create table app_admins (
  id            bigint generated always as identity primary key,
  name          text not null,
  access_token  text unique not null default gen_random_uuid()::text,  -- 마스터 토큰
  created_at    timestamptz default now()
);
```

- 마스터 토큰을 가진 링크는 **모든 사업장** 수정 가능. 이 토큰은 외부 노출에 특히 주의.

### 3-3. sites (사업장, 59개소) — 사업장 단위 정보 포함

> **설계**: 완료여부·설치일·담당자명·비고는 **사업장 단위로 한 번만** 기록한다.
> (품목별로 나누지 않음)

```sql
create table sites (
  id             bigint generated always as identity primary key,
  name           text not null,     -- 예: "1번 사업장"
  contractor_id  bigint references contractors(id),  -- 이 사업장 담당 업체

  -- 아래는 사업장 단위 입력 항목 (담당자가 입력)
  status         text default '미착수'          -- 완료여부: 미착수 / 진행중 / 완료
                   check (status in ('미착수','진행중','완료')),
  install_date   date,              -- 설치일
  manager_name   text,              -- 담당자명
  note           text,              -- 비고

  updated_by     text,              -- 마지막 수정자 (자동 기록)
  updated_at     timestamptz default now(),  -- 마지막 수정 시각 (자동 기록)
  created_at     timestamptz default now()
);
```

### 3-4. installations (사업장 × 품목별 수량/위치)

> 품목마다 다른 건 **수량**과 **설치위치**뿐이라, 이 두 가지만 품목별로 둔다.

```sql
create table installations (
  id           bigint generated always as identity primary key,
  site_id      bigint references sites(id) not null,
  product      text not null check (product in ('A','B','C')),
  planned_qty  integer not null default 0,   -- 예정수량: 고정값, 화면에서 수정 불가
  actual_qty   integer,                      -- 실제 설치수량: 담당자 입력
  location     text,                         -- 설치위치: B, C 품목만 사용 (A는 미사용/무시)
  unique (site_id, product)
);
```

- **A 품목**: 정해진 위치가 있어 설치위치 입력란 없음 → `location` 사용 안 함.
- **B, C 품목**: 설치위치 입력란 있음 → `location` 사용.

## 4. 권한 모델 (URL 토큰 + RLS + RPC)

### 핵심 아이디어

- **조회(SELECT)**: 모두 허용. 현황판을 누구나 함께 본다.
- **수정(UPDATE)**: anon 키로 테이블에 직접 수정하는 건 **막는다**.
  대신 저장 전용 **DB 함수(RPC)** 를 통해서만 수정하고, 그 함수 안에서 **토큰을 검증**한다.
  → 자기 업체가 맡은 사업장(또는 마스터)일 때만 실제로 반영된다.
  이렇게 하면 UI뿐 아니라 **서버단에서 진짜로** 권한이 걸린다.

### RLS 설정 (조회만 열고, 직접 수정은 차단)

```sql
alter table sites enable row level security;
create policy "sites_read_all" on sites for select using (true);

alter table installations enable row level security;
create policy "inst_read_all" on installations for select using (true);

-- UPDATE/INSERT 정책을 anon 에게 주지 않음 → 직접 수정 불가.
```

### 저장 함수 (RPC) — 여기서 토큰 검증

```sql
-- 토큰이 해당 사업장을 수정할 권한이 있는지 판정하는 헬퍼
create or replace function can_edit_site(p_token text, p_site_id bigint)
returns boolean language sql stable as $$
  select
    exists (select 1 from app_admins where access_token = p_token)  -- 마스터면 전체 허용
    or exists (
      select 1 from sites s
      join contractors c on c.id = s.contractor_id
      where s.id = p_site_id and c.access_token = p_token
    );
$$;

-- 수정자 이름 조회 헬퍼
create or replace function who_is(p_token text)
returns text language sql stable as $$
  select coalesce(
    (select name from contractors where access_token = p_token),
    (select '[관리자] ' || name from app_admins where access_token = p_token)
  );
$$;

-- 사업장 단위 항목 저장
create or replace function save_site(
  p_token text, p_site_id bigint,
  p_status text, p_install_date date, p_manager_name text, p_note text
) returns void language plpgsql security definer as $$
begin
  if not can_edit_site(p_token, p_site_id) then
    raise exception '이 사업장을 수정할 권한이 없습니다';
  end if;
  update sites set
    status = p_status, install_date = p_install_date,
    manager_name = p_manager_name, note = p_note,
    updated_by = who_is(p_token), updated_at = now()
  where id = p_site_id;
end; $$;

-- 품목별 항목 저장 (실제수량, 설치위치)
create or replace function save_installation(
  p_token text, p_site_id bigint, p_product text,
  p_actual_qty integer, p_location text
) returns void language plpgsql security definer as $$
begin
  if not can_edit_site(p_token, p_site_id) then
    raise exception '이 사업장을 수정할 권한이 없습니다';
  end if;
  update installations set
    actual_qty = p_actual_qty,
    location = case when p_product in ('B','C') then p_location else null end
  where site_id = p_site_id and product = p_product;
end; $$;
```

- `security definer` 덕분에 함수는 RLS를 통과해 UPDATE 하지만, 그 전에 토큰을 검증하므로
  권한 없는 저장은 예외로 막힌다.
- 프론트엔드는 저장 시 `supabase.rpc('save_site', {...})` / `supabase.rpc('save_installation', {...})`
  를 호출하고, `p_token` 에는 URL에서 읽은 토큰을 넣는다.

### 접속 흐름

- 링크: `https://<깃허브페이지주소>/?token=<토큰>`
- 페이지 로드 시 URL에서 토큰을 읽어 → 그 토큰이 담당하는 사업장을 파악.
- 자기 담당 사업장은 입력칸 활성화, 나머지는 읽기 전용.
- 토큰이 없거나 잘못되면 → 전체 조회만 가능(수정 불가).

> **주의(토큰 방식의 한계)**: 링크가 유출되면 그 토큰 범위의 사업장은 남이 수정할 수 있다.
> 특히 마스터 토큰은 절대 외부 공유 금지. "실수 방지" 목적엔 충분하나, 완전한 보안이 필요하면
> 다음 단계로 매직링크(Supabase Auth) 로그인으로 업그레이드.

## 5. 화면 구성 (UI)

### 5-1. 전체 현황 대시보드 (기본 화면 / 누구나 조회)

- 59개소 사업장 목록을 표 또는 카드로 표시.
- 각 사업장별로 A/B/C 품목의 **예정수량 / 실제수량**, 그리고 사업장의 **완료여부**를 한눈에.
- 전체 진행률(예: 완료 사업장 수 / 전체 사업장 수) 요약 표시.
- 사업장명 또는 업체로 검색·필터.
- URL 토큰으로 접속한 담당자에게는 "내 담당 사업장" 을 상단에 강조.

### 5-2. 사업장 상세 / 입력 화면

한 사업장을 클릭하면 진입.

**(1) 사업장 단위 항목 — 한 번만 입력** → `save_site` RPC 호출

| 항목 | 입력 형태 | 편집 권한 |
|------|-----------|-----------|
| 완료여부 | 미착수 / 진행중 / 완료 선택 | 담당자·마스터 |
| 설치일 | 날짜 선택 | 담당자·마스터 |
| 담당자명 | 텍스트 | 담당자·마스터 |
| 비고 | 텍스트 | 담당자·마스터 |

**(2) 품목별 항목 — A / B / C 각각** → `save_installation` RPC 호출

| 항목 | A 품목 | B 품목 | C 품목 | 편집 권한 |
|------|:---:|:---:|:---:|-----------|
| 예정수량 (읽기전용·고정) | O | O | O | 수정 불가 |
| 실제 설치수량 (숫자 입력) | O | O | O | 담당자·마스터 |
| 설치위치 (텍스트) | **없음** | O | O | 담당자·마스터 |

- 자기 담당이 아닌 사업장에 들어오면 모든 입력칸이 **비활성(읽기 전용)** 으로 보여야 함.
- 저장 시 `updated_by`(수정자), `updated_at`(시각) 자동 기록 → "누가 언제" 추적.

### 5-3. 관리자(마스터) 화면

- 마스터 토큰으로 접속 시: 전체 사업장 목록 + 모든 항목 수정 가능.
- 업체–사업장 배정 현황(어느 업체가 어느 사업장 담당인지) 확인.
- 전체 데이터 CSV/엑셀 내보내기(있으면 좋음).

## 6. 초기 데이터 (시드)

> **아직 미확정**: 실제 사업장명, 담당업체(4~6곳), 품목별 예정수량은
> 사용자가 "59개 사업장 목록" 파일을 주면 채운다.
> 그전까지는 아래 placeholder 로 만든다.

- 사업장: `1번 사업장` ~ `59번 사업장`
- 각 사업장마다 품목 A, B, C 행 3개 (총 59 × 3 = 177개 installations 행)
- `planned_qty`(예정수량)는 목록 받기 전까지 `0` 으로 둔다.

placeholder 생성 SQL 예시:

```sql
-- 59개 사업장 생성
insert into sites (name)
select n || '번 사업장' from generate_series(1,59) as n;

-- 각 사업장에 A/B/C 품목 행 생성 (예정수량 0으로 초기화)
insert into installations (site_id, product, planned_qty)
select s.id, p.product, 0
from sites s
cross join (values ('A'),('B'),('C')) as p(product);

-- 마스터 관리자 1명 생성 (토큰 자동 발급)
insert into app_admins (name) values ('마스터');

-- 업체 예시 (실제 목록 받으면 이름 교체 / 개수 4~6개로)
insert into contractors (name)
values ('업체1'),('업체2'),('업체3'),('업체4');

-- 발급된 토큰 확인 (이 값으로 링크를 만든다)
select name, access_token from app_admins;
select name, access_token from contractors;
```

목록을 받은 뒤:
- 사업장명, 담당업체(`sites.contractor_id` 연결), 품목별 예정수량을 UPDATE 로 채운다.
- 목록은 아래 형태의 CSV 로 받으면 그대로 반영 가능:

```
사업장명, 담당업체, A_예정수량, B_예정수량, C_예정수량
1번 사업장, 가나다전기, 10, 5, 3
...
```

## 7. 배포 절차 (초보 기준 단계별)

1. **Supabase 프로젝트 생성** (무료) → project URL, anon key 확보.
2. SQL Editor 에서 3장·4장의 테이블/함수/정책 SQL 실행.
3. 6장 시드 SQL 실행 → 사업장/업체/마스터 생성, **토큰 확인**.
4. 프론트엔드 코드 작성 (5장 화면). 저장은 RPC(`save_site`, `save_installation`) 호출.
5. GitHub 저장소 생성 → 코드 push.
6. GitHub **Settings → Pages** 에서 배포 켜기 → 공개 링크 생성.
7. 프론트엔드에 Supabase URL·anon key 연결.
8. 업체별 링크(`?token=...`)와 마스터 링크를 각 담당자에게 배포.
